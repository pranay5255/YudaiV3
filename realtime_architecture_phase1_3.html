Scope & Decision Snapshot

These are the key architectural constraints locked by the questionnaire and reflected in the phase plan. They drive nearly every diagram below.

Implemented in code Partial / shell / integration missing Planned by docs, not fully implemented

Core MVP Decisions (Confirmed)

  • Sandbox identity key is org + repo + environment
  • One repo, two entrypoints first (controller + sandbox server)
  • Direct tunnel from frontend to sandbox (no proxy fallback)
  • Streaming split is SSE for solver trajectory and WebSocket for chat
  • PostgreSQL stores metadata; sandbox stores append-only JSON cache under /home/yudai/.cache/
  • Session persistence ends only after both GitHub issue creation and PR creation

Operational Constraints

  • Sandbox created at session creation
  • Liveness probe / heartbeat every 10s
  • Tunnel auth reuses session JWT (TTL 1h, reusable)
  • CORS target for tunnel-facing server: https://yudai.app
  • Single active editor semantics in Phase 1
  • Mandatory yudai-grep load at sandbox boot is a Phase 2 goal (not fully enforced yet)

Phase 1 Session Provisioning Flow (Controller + Tunnel)

This is the core Phase 1 flow implemented in the store/controller/lifecycle stack: creating a session, ensuring a runtime, provisioning or reusing a sandbox, and returning a tunnel URL to the frontend. 

sequenceDiagram
  autonumber
  participant User
  participant FE as Frontend (sessionStore)
  participant API as Session API (/daifu/sessions)
  participant CTR as Controller (/controller/sessions/:id/runtime)
  participant L as RealtimeLifecycleService
  participant DB as PostgreSQL
  participant SM as SandboxManager

  User->>FE: Select repo + branch
  FE->>API: POST /daifu/sessions (Bearer session token)
  API->>DB: Create ChatSession
  API-->>FE: Session created (session_id)

  FE->>CTR: POST /controller/sessions/SESSION_ID/runtime
  CTR->>DB: Load ChatSession (owned by current user)
  CTR->>L: create_runtime_for_session(...)
  L->>DB: Resolve sandbox by identity_key (org:owner/repo:env)
  alt Existing active sandbox for same identity and same session
    L->>DB: Reuse sandbox + runtime
  else No sandbox / terminated sandbox
    L->>DB: Insert Sandbox (provisioning -> running)
    L->>SM: build_tunnel_url(sandbox_id)
    L->>SM: ensure_git_bootstrap(clone once / periodic fetch)
    L->>DB: Insert SessionRuntime
  else Active sandbox owned by another session
    L-->>CTR: SINGLE_ACTIVE_EDITOR_CONFLICT (409 hard error)
  end
  L->>DB: Insert SessionAuditEvent (sandbox_start)
  L->>L: Append cache event in /home/yudai/.cache/session/*.json
  L->>SM: Start probe task (best-effort) for /healthz
  CTR-->>FE: runtime + tunnel_url + token TTL

  FE->>FE: Route future session HTTP calls to tunnel target

Implemented (Phase 1 foundation)

  • Controller runtime ensure/get endpoints
  • Sandbox create/get/delete/resolve-tunnel/heartbeat/cleanup endpoints
  • Identity normalization and single-active-editor conflict handling
  • Session runtime persistence and audit event writes
  • Frontend store can request runtime and reroute session calls to tunnel URL

Important MVP Behavior

  • No proxy fallback when tunnel is unavailable (hard error path required)
  • Tunnel auth reuses current session JWT (frontend bearer token / SSE query token)
  • Tunnel URL is currently a templated value from env (SANDBOX_TUNNEL_TEMPLATE)

Completion / Export / Termination Flow (Issue + PR criteria)

The questionnaire defines session completion as both GitHub issue creation and PR creation. The lifecycle service already implements this completion detector and termination pipeline. 

sequenceDiagram
  autonumber
  participant FE as Frontend
  participant S as Sandbox/Session API
  participant GH as GitHub
  participant SOL as Solver Manager
  participant L as RealtimeLifecycleService
  participant DB as PostgreSQL
  participant C as SessionCacheStore

  FE->>S: Create GitHub issue from session issue
  S->>GH: Create Issue
  GH-->>S: issue_url + issue_number
  S->>L: mark_issue_created(session_id, issue_url, issue_number)
  L->>DB: Update SessionRuntime.completion_issue_created = true
  L->>DB: Insert audit event github_issue_create
  L->>C: Append cache event + merge github_refs(issue)
  L->>L: finalize_on_completion()? (waits for PR)

  FE->>S: Start solve (issue_id)
  S->>SOL: start_solve(...)
  SOL->>L: record_solve_start(...)
  L->>DB: Insert audit event solve_start
  L->>C: Append cache event

  SOL->>GH: Create PR (during successful run)
  GH-->>SOL: pr_url
  SOL->>L: mark_pr_created(session_db_id, pr_url, pr_number)
  L->>DB: Update SessionRuntime.completion_pr_created = true
  L->>DB: Insert audit event pr_create
  L->>C: Append cache event + merge github_refs(pr)

  L->>L: _finalize_on_completion() now true (issue && PR)
  L->>DB: Set completion_detected + completion_reason + completed_at
  L->>DB: Collect SolveRun trajectory refs
  L->>C: Merge trajectory refs
  L->>C: Export bundle (tar.gz + metadata JSON)
  L->>DB: Insert session_artifacts row
  L->>DB: Terminate sandbox + runtimes + audit event sandbox_terminate
The artifact export path already writes bundle metadata and checksums and stores a corresponding row in session_artifacts. Object storage (S3) is represented in metadata shape but actual upload/integration remains a deployment/phase-hardening task.

Open questions

This section focuses on still-untested or not fully validated areas in the current codebase, not just theoretical questions from the docs.

High-priority untested/partial areas

  • End-to-end direct tunnel behavior in a real deployment: controller returns tunnel URL, but production tunnel DNS/TLS/proxy rules and browser CORS behavior have not been fully validated against the documented no-fallback policy.
  • Browser validation gates from Phase 1: the plan requires sign-in → session create → sandbox provision → issue → solve → PR → auto-terminate → artifact row verification. Code paths exist, but full browser smoke evidence is still needed.
  • WS chat protocol (Phase 3): sandbox WS endpoint currently authenticates and echoes. The final message schema (plain/token/tool/status), reconnect handling, and backpressure warnings are not implemented/tested end-to-end.
  • SSE final Phase 3 semantics: current SSE stream works, but the documented 3s heartbeat + 10s forced reconnect + explicit terminal contract in sandbox server is not fully implemented or validated.
  • Mandatory yudai-grep boot behavior (Phase 2): current integration is optional/lazy. Hard-fail startup and diagnostics endpoint behavior are not wired/tested.
  • Training pipeline from exported trajectories (Phase 2): yudai-grep training utilities exist, but no admin CLI currently reads session_artifacts metadata + exported cache to build a real dataset from session trajectories.
  • Cleanup and liveness operations in production: liveness probe logic exists, but the lifecycle of probe tasks and periodic cleanup execution in deployed controller processes still needs staging validation.
  • Sandbox git bootstrap on real repos: clone/fetch policy exists but needs validation for auth/private repos, branch mismatch, and failure handling under deployment conditions.

Questions to answer during validation (not necessarily design blockers)

  • Which host/service actually provisions sandbox compute in production (today the code builds/records tunnel URLs and lifecycle state, but provisioning orchestration is still mostly local/service-template oriented)?
  • Should SSE remain in backend/solver/solver.py or be fully moved behind sandbox-specific realtime routes for a cleaner Phase 3 contract?
  • What exact observability surface is required for runtime/sandbox state debugging (metrics, traces, audit dashboard, admin endpoints)?
  • How will object storage upload be integrated for artifact bundles (current metadata includes S3-shaped references, but no uploader is wired)?

Deployment Changes

These are the key things to get right or complete to ship the final split real-time deployment safely.

1) Deploy the split topology explicitly

  • Deploy controller host and sandbox session server as separate services/process roles (even if built from one repo/image).
  • Keep backend/run_server.py only for local/dev compatibility unless intentionally used in production.
  • Define health checks for both services (/health and /healthz).

2) Get tunnel routing + TLS + CORS correct

  • Provision a stable tunnel domain pattern that matches SANDBOX_TUNNEL_TEMPLATE and real sandbox addressing.
  • Ensure sandbox server CORS allows https://yudai.app exactly (per questionnaire).
  • Confirm frontend receives usable public tunnel URLs (not internal-only URLs).
  • Enforce no hidden proxy fallback in frontend and edge configs.

3) Wire required environment variables and secrets

  • DATABASE_URL / CONTROLLER_DATABASE_URL
  • Realtime flags: REALTIME_* / VITE_REALTIME_*
  • CONTROLLER_BASE_URL, SANDBOX_ID, CONTROLLER_HEARTBEAT_SECRET
  • GitHub OAuth / app credentials and model provider secrets
  • Optional future: S3 credentials for artifact upload (currently metadata-only representation)

4) Run DB migrations and rehearse rollback

  • Apply the realtime lifecycle tables/indexes migration (already present in SQL files).
  • Validate rollback scripts in staging before production rollout.
  • Confirm indexes support lifecycle queries under expected traffic.

5) Persist and export sandbox cache artifacts safely

  • Ensure sandbox has writable storage for /home/yudai/.cache/.
  • Decide whether artifact bundles remain local-only temporarily or are uploaded to S3 during completion flow.
  • If using S3, implement upload + retry + checksum verification before marking artifact metadata complete.

6) Add production jobs/loops for cleanup and observability

  • Run a periodic controller cleanup job calling /controller/sandboxes/cleanup (or internal scheduler equivalent).
  • Centralize logs for sandbox lifecycle, heartbeat failures, stream disconnects, and audit events.
  • Add dashboards/alerts for terminated-on-completion rate, tunnel errors, heartbeat misses, and SSE/WS failures.

7) Complete Phase 2 and Phase 3 release-critical gaps

  • Phase 2: enforce mandatory yudai-grep load at sandbox boot + diagnostics endpoint + admin training CLI.
  • Phase 3: finalize WS chat protocol and frontend WS client/reconnect UI (10 retries), plus SSE final contract and forced reconnect behavior.

8) Validate frontend production build/runtime behavior

  • Confirm realtime feature flags are correctly injected at build time.
  • Verify direct tunnel routing works after auth refresh and session reload.
  • If React 19 upgrade is part of this release, run full typecheck/build/test/browser smoke on the upgraded frontend before cutover.

9) Execute the browser validation gates from the plan

  • Run the P1/P2/P3 browser gates in staging exactly as listed in REAL_TIME_PHASE_TASK_LIST.md.
  • Record evidence (screenshots/logs/DB rows/artifact files) for each gate before enabling flags in production.
