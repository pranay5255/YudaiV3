2 Option to replace sandbox session server with zeroclaw / picoclaw / nanoclaw ▾
Design Decision Phase 1 Architectural

Current Architecture

The sandbox session server (backend/run_sandbox_server.py) is a FastAPI application that hosts:

  • Session routes — chat, issues, context cards, file deps, messages
  • Sandbox routes — healthz, WS chat shell
  • Solver routes — start/status/cancel/stream
  • Heartbeat loop — reports back to controller every 10s

The Pluggable Agent Pattern

To make the sandbox server replaceable by an alternative agent (zeroclaw, picoclaw, nanoclaw), you need a contract boundary between the controller and whatever runs inside the sandbox.

The contract is already mostly defined:

The controller only cares about: (1) /healthz responding OK, (2) the sandbox accepting the tunnel URL, (3) the heartbeat mechanism. Everything else is internal to the sandbox.

4 Diagram 03: Session API as entrypoint — but it's inside the sandbox? ▾
Phase 1 Critical Clarification

The Confusion

Diagram 03-phase1-session-provisioning-flow.md shows: 

User -> FE -> Session API (POST /daifu/sessions) -> DB -> ... session created
FE -> Controller (POST /controller/sessions/ID/runtime) -> ... provisions sandbox

You're asking: "If the Session API is inside the sandbox, how can it be hit first? The sandbox doesn't exist yet!"

The Answer: The Session API is on BOTH sides

Key insight: Session CREATION happens on the controller host, not inside the sandbox.

The session CRUD endpoint (POST /daifu/sessions) is mounted on run_controller.py (line 54). After the session record exists in PostgreSQL, the frontend calls the controller to provision the sandbox runtime. Only AFTER the sandbox is running does the frontend switch to the tunnel URL for subsequent session operations.

7 Complete WS chat endpoint — bare implementation ▾
Phase 3 Implementation

Current State

backend/realtime/sandbox_routes.py has a WS endpoint that:

  • Authenticates via JWT query param
  • Sends a status: connected message
  • Echoes all received messages back

It does not: process chat through ChatOps, stream token chunks, emit tool events, or handle errors.

Bare WS Chat Implementation Plan

Keep it simple. The WS endpoint should wrap the existing ChatOps.process_chat_message() and stream the response.

Message Types (Inbound from Client)

TypePayload
chat_message{ text, context_card_ids?, repository? }
ping{}

Message Types (Outbound to Client)

TypeWhenPayload
statusConnection established, state changes{ status, session_id, message }
chat_replyFull chat response ready{ message_id, text, actions?, processing_time }
token_chunkStreaming tokens (future LLM streaming){ chunk, message_id, index }
tool_eventTool invocation during chat (future){ tool_name, status, result? }
errorProcessing failure{ code, message, recoverable }

Minimal Implementation Structure

@router.websocket("/sessions/{session_id}/ws/chat")
async def websocket_chat(websocket, session_id, token, db):
    # 1. Auth (existing)
    user = validate_session_token(db, token)
    if not user: close(4401); return

    # 2. Accept + status
    await websocket.accept()
    send_json(type="status", status="connected")

    # 3. Message loop
    while True:
        msg = await websocket.receive_json()

        if msg["type"] == "ping":
            send_json(type="pong")
            continue

        if msg["type"] == "chat_message":
            try:
                send_json(type="status", status="processing")

                # Call existing ChatOps (same as REST endpoint)
                response = ChatOps.process_chat_message(
                    session_id, msg["text"],
                    msg.get("context_card_ids", []),
                    msg.get("repository"), db, user
                )

                send_json(type="chat_reply",
                    message_id=response.message_id,
                    text=response.reply,
                    actions=response.actions,
                    processing_time=response.processing_time)

            except Exception as e:
                send_json(type="error",
                    code="CHAT_PROCESSING_FAILED",
                    message=str(e),
                    recoverable=True)
Note on future nanoclaw/zeroclaw refactor (Q2):

If you replace the chat backend with an agent like nanoclaw, the WS endpoint stays the same — you just swap ChatOps.process_chat_message() for agent.handle_message(). The message schema is agent-agnostic.

Frontend Changes Required

src/components/Chat.tsx currently calls sendChatMessage() via REST. Replace with a WS client hook:

The Two Terminal Tasks

You described two tasks the sandbox chat/session API can instruct:

  1. Create a GitHub issue on the local cloned repo — this is the first terminal task
  2. Solve that GitHub issue — second terminal task, produces a PR

Both must complete to trigger session completion (questionnaire Q33).

Current Solve Pipeline

System Design Problems Highlighted

Problem 1: SSE Route Ownership

SSE stream is in solver.py which is mounted on the session router, not the sandbox router.

This means in the split architecture, the SSE endpoint is available on both controller AND sandbox server (because both mount session_router). The controller should NOT serve SSE streams — only the sandbox should. Fix: Move SSE to sandbox_routes.py or create a dedicated sandbox_solver_routes.py.

Problem 2: 10s Max Stream Duration

The questionnaire says max SSE stream duration is 10s before forced reconnect (Q55). But solver trajectories can run for minutes. This means:

  • The SSE stream must chunk into 10s segments
  • Each segment ends with a reconnect_required event
  • Frontend auto-reconnects and resumes from where it left off
  • This is effectively a long-poll-over-SSE pattern

Problem 3: WS/REST Coexistence During Transition

During the Phase 3 transition, both REST chat and WS chat will exist.

Use the feature flag REALTIME_WS_CHAT_ENABLED to switch. Frontend should check realtimeFeatureFlags.wsChatEnabled and choose between sendChatMessage() (REST) and useChatWebSocket (WS). Both must work simultaneously until WS is validated.

10 Open Questions: Deep Analysis of System Design Impact ▾
System Design Critical

Question 1: Which host/service actually provisions sandbox compute in production?

Impact: HIGH — This is the single biggest unresolved question.

Current state: The code in lifecycle.py builds tunnel URLs and records lifecycle state, but the actual provisioning call (Modal API Sandbox.create(), container image pull, tunnel assignment) is represented by placeholder logic. The SandboxManager methods build_tunnel_url() and ensure_git_bootstrap() are called but don't make real Modal API calls yet.

Design impact:

  • Startup latency: Modal sandbox cold start is ~10-30s (questionnaire says target 10s). If provisioning is slow, session create will block the frontend.
  • Tunnel URL format: Modal assigns tunnel URLs in a specific format. The controller currently generates synthetic URLs. Production URLs will look different.
  • Error handling: Modal provisioning can fail (rate limits, image pull failures, resource exhaustion). The lifecycle service needs retry logic or clear error propagation.
  • Git bootstrap: Cloning a private repo inside Modal requires auth token injection at sandbox start (questionnaire Q81: env vars only at start). Token must be available before sandbox boots.

Answer: For MVP, the controller's create_runtime_for_session() must make a real Modal API call. Build a thin ModalSandboxProvider class that wraps modal.Sandbox.create() with the image, env vars, and tunnel configuration. This is a Phase 1 hard blocker.

Question 2: Should SSE remain in solver.py or move to sandbox-specific routes?

Impact: MEDIUM — Affects route ownership and Phase 3 contract clarity.

Current state: SSE stream is in backend/solver/solver.py and mounted via session_router. Because run_controller.py also mounts session_router, the SSE endpoint is accidentally available on the controller host.

Design impact:

  • If a user hits the SSE endpoint on the controller instead of the sandbox, they'll get a broken stream (no executor running on controller).
  • The Phase 3 contract calls for the SSE endpoint to live inside the sandbox server only.
  • Moving it to sandbox_routes.py makes ownership explicit.

Answer: Move SSE to sandbox routes. The solver execution logic stays in solver.py but the streaming endpoint should be in sandbox_routes.py or a new sandbox_solver_routes.py. The controller should not expose solver/stream routes.

Question 3: What observability surface is required?

Impact: LOW for MVP — but accumulates technical debt.

Current state: Questionnaire says "not required for now in mvp, manual monitoring" (Q86, Q87). The audit event table exists but no dashboard, no metrics export, no structured logging.

Design impact for later:

  • Without structured logs, debugging production sandbox failures will be painful.
  • Without metrics, you won't know sandbox startup latency, chat response times, or solve success rates.
  • The audit_events table is the minimum viable observability surface.

Answer: For MVP, rely on session_audit_events table + application logs. Add a simple GET /admin/audit-events endpoint for querying. Defer Prometheus/Grafana/etc.

Question 4: How will S3 upload be integrated for artifact bundles?

Impact: MEDIUM — Artifacts are lost on sandbox termination without this.

Current state: cache_store.py exports bundles to sandbox local filesystem. Metadata is written to PG with S3-shaped fields. No actual S3 client is wired.

Design impact:

  • When _finalize_on_completion() terminates the sandbox, the .tar.gz bundle on sandbox FS is destroyed.
  • The session_artifacts.bundle_path in PG will point to a non-existent file.
  • Training pipeline (Phase 2) reads from exported trajectories — if they're gone, training has no data.

Answer: Three options for MVP:

  1. Best: Add S3 upload in _finalize_on_completion() before termination. Simple boto3 call.
  2. Acceptable: Copy bundle to controller host via HTTP before termination (controller provides an artifact upload endpoint).
  3. Minimal: Keep bundle in PG as a BYTEA column (works for small artifacts only).

Critical CORS issue:

Questionnaire Q24 sets CORS to https://yudai.app. But the sandbox server is accessed via Modal's tunnel URL (e.g., https://sandbox-xxx.modal.run). The browser will make cross-origin requests from yudai.app to sandbox-xxx.modal.run. The sandbox server must set CORS to allow https://yudai.app as origin. Currently run_sandbox_server.py has allow_origins=["*"] in dev — this needs to be locked down to https://yudai.app in production. WebSocket connections are NOT subject to CORS preflight but the browser still sends Origin header which the server should validate.

Area 3: WS Chat Protocol (Phase 3)

What exists: Auth + echo shell.

What's missing:

  • Message type dispatcher (plain/token/tool/status)
  • ChatOps integration (actual chat processing)
  • Backpressure handling (buffer + bounded queue + drop oldest, per questionnaire Q60)
  • Reconnect handling (server-side: clean disconnect with reason code)
  • 10-retry client with exponential backoff

Design impact: Without this, Phase 3 cannot pass browser validation gate P3-B1.

Area 4: SSE Final Phase 3 Semantics

What exists: SSE stream in solver.py with trajectory polling.

What's missing:

  • 3s heartbeat: Current implementation polls every 3s but doesn't send explicit heartbeat SSE events between data events. It should send event: heartbeat\ndata: {}\n\n even when no trajectory update is available.
  • 10s forced reconnect: Stream runs indefinitely. Need asyncio.wait_for(generator, timeout=10) wrapper that emits event: reconnect_required then closes.
  • Terminal events: done event exists but cancelled and failed are not consistently emitted.

Area 7: Cleanup and Liveness in Production

What exists: lifecycle.py has cleanup_stale_sandboxes() and probe task management.

What's untested:

  • Probe tasks as asyncio background tasks — do they survive across request cycles?
  • Cleanup scheduler — who calls cleanup_stale_sandboxes() periodically? Need a cron/scheduler in the controller.
  • What happens when the controller restarts? Orphaned sandboxes need discovery and cleanup.

Area 8: Sandbox Git Bootstrap on Real Repos

What exists: Clone once + periodic fetch policy (questionnaire Q29).

What's untested:

  • Private repos: Need auth token in git config. Token must be injected via env var at sandbox start.
  • Large repos: Clone time could exceed sandbox startup SLO (10s). May need shallow clone.
  • Branch mismatch: User selects branch "feature-x" but it doesn't exist in the remote. Error handling needed.
  • Periodic fetch interval: Not defined. Too frequent = noisy, too rare = stale.
