Core Pain Points
  • Cold sandbox per solve: Each issue solve creates a fresh Modal sandbox, clones the repo, installs dependencies. ~30s startup before any agent work begins.
  • No persistent context: Sandbox dies after solve completes. All codebase understanding, embeddings, and file indexes are discarded. The next solve starts from zero.
  • Polling-based streaming: SSE endpoint polls executor every 3 seconds for trajectory updates. Not true real-time — creates perception lag and wastes resources.
  • Session and solver are decoupled: Chat sessions (session_routes.py) and solver runs (solver.py) share a session_id but the solver doesn't inherit session context. The user's conversation context doesn't flow into the agent.
  • yudai-grep is dormant: The ML model for intelligent file routing exists but has no trained checkpoint. Its integration in facts_and_memories.py always falls back to empty results.

3 — The Modal Vibe Pattern
What Modal Vibe Teaches Us

Modal Vibe (github.com/modal-labs/modal-vibe) demonstrates a pattern where each user gets a persistent sandbox with a direct tunnel. The key insight: sandboxes are not batch workers — they are long-lived servers that users interact with directly.

Persistent Sandboxes
Each app lives in its own Modal Sandbox with a webserver accessible through Modal Tunnels. Sandboxes persist across interactions.
Direct Tunnel Access
Frontend discovers tunnel_url from manager and communicates directly with sandbox. No proxy bottleneck.
Manager Orchestrates
FastAPI controller creates, monitors, and cleans up sandboxes. Health checks every 60s terminate unresponsive ones.
Scale-Tested
1 app ~30s, 100 apps ~150s, 1000 apps ~300s. 90% of latency is LLM generation, not sandbox creation.

4 — Proposed Architecture
Core Idea

Wrap the entire sessions API inside a persistent Modal Sandbox per repository. Instead of the sandbox being a disposable worker for individual solves, each sandbox becomes a long-lived codebase server that holds the repo clone, the yudai-grep model, file indexes, conversation context, and the sessions API itself. Users talk to their codebase through this sandbox in real-time via direct tunnel.

5 — yudai-grep's Role in Real-Time Sessions
From Dormant to Core Infrastructure

In the new architecture, yudai-grep moves from an optional import with a try/except fallback to a core component loaded at sandbox boot. It becomes the intelligent file router that makes real-time interaction possible.

Query Router
Every user message passes through yudai-grep first. The model predicts which files to read, grep, glob, or summarize — eliminating slow full-repo scans.
Context Budgeting
With limited LLM context windows, yudai-grep ranks files by relevance probability. Only the top-K files enter the prompt — maximizing signal-to-noise ratio.
Trajectory Learning
Each solve and chat interaction produces a trajectory. These become training data for yudai-grep — the model improves per-repo over time without manual curation.
Tool Selection
Predicts not just which files but how to access them. read for full context, grep for pattern search, glob for discovery, summarize for overview.

Phase 4: Advanced Features (future)
  • Multi-user sessions: Multiple developers connected to the same sandbox via shared tunnel. Cursor-style collaboration.
  • Parallel solver runs: Sandbox can spawn sub-sandboxes for parallel model evaluation. max_parallel becomes configurable.
  • Persistent embeddings: pgvector indexes stored in sandbox filesystem. Survives across snapshots.
  • IDE integration: VSCode extension that connects to sandbox tunnel for inline code suggestions.
  • Repo-specific fine-tuning: Accumulated trajectories feed into LoRA fine-tuning for per-repo model specialization.

The recommendation: Start with Phase 1 — split the monolith into controller + sandbox. This unblocks everything else. The controller becomes a thin auth + orchestration layer (~300 lines). The sandbox gets the full sessions API, repo clone, and yudai-grep. Frontend switches from proxied /api/* to direct tunnel_url access. Each subsequent phase builds on this foundation without requiring further architectural changes.
