Shared Session Streaming

yudai.app/session/{slug} — live agent trajectory for viewers

Overview

Build a shareable, read-only live view of an agent run. An owner creates a session, gets a link like yudai.app/session/bananarepublic, and shares it. Viewers open the link and see the agent's trajectory streamed in real-time from the Modal sandbox.

1 — What Changes from the Network Diagram

The existing networking infrastructure handles everything. No new servers, ports, DNS records, or Caddy config. The only change is one new WebSocket route inside FastAPI that piggybacks on the existing :443 → Caddy → :8000 pipeline.

Traffic path is identical for viewers Viewer's browser → DNS resolves api.yudai.app to 139.84.154.9 → Caddy on :443 terminates TLS → forwards to localhost:8000 → FastAPI handles the /share/{slug}/ws WebSocket upgrade. Same ports, same proxy, same cert.

DNS

No changes. yudai.app still points to Vercel. api.yudai.app still points to 139.84.154.9. /session/* is a client-side route handled by React Router inside the SPA.

Caddy

No changes to /etc/caddy/Caddyfile. The reverse_proxy localhost:8000 rule already forwards all paths including /share/*. WebSocket upgrade is handled automatically.

Firewall (UFW)

No changes. Only :22, :80, :443 open. All viewer traffic enters through :443 like everything else.

CORS

No changes. ALLOW_ORIGINS=https://yudai.app already permits the frontend origin. Viewer's browser makes requests from the same origin.

2 — Share Link Generation Flow

The owner creates a shareable link from their active agent session.

1

Owner starts an agent session

Creates ChatSession + SessionRuntime + Modal sandbox. The existing unified WebSocket streams trajectory to the owner's browser.

2

Owner clicks "Share"

Frontend calls POST /sessions/{sessionId}/share with the owner's Bearer token.

3

Backend creates a SessionShare record

Generates a unique slug (e.g., bananarepublic) and a share_token. Stores in the session_shares table with the session_id and permissions.

4

Backend returns the share URL

Returns { slug: "bananarepublic", share_url: "https://yudai.app/session/bananarepublic" }

5

Owner copies and shares the link

The link can be sent via Slack, Discord, email, etc. Anyone with the link can view the session.

3 — Viewer Connection Flow

What happens when someone opens a shared session link.

1

Viewer opens https://yudai.app/session/bananarepublic

Browser resolves yudai.app → 76.76.21.21 (Vercel CDN). Vercel serves the React SPA bundle — the same index.html + index.js as always.

2

React Router matches /session/:slug

Renders the new <SharedSessionPage /> component. No auth required to load this page.

3

Component calls GET https://api.yudai.app/share/bananarepublic

No auth header needed. Backend resolves slug → session_id, checks is_active and expires_at, returns session metadata (title, current mode, repo info) + a temporary viewer_token.

4

Component opens WebSocket

wss://api.yudai.app/share/bananarepublic/ws?token={viewer_token}
Backend validates token, registers the WS as a read-only listener on the session's SessionWebSocketHub.

5

Viewer sees live trajectory

The hub broadcasts trajectory_update, llm_stream, sandbox_stream, tool_call, state_event events to all listeners — including the viewer. The viewer's UI renders the trajectory in real-time using the existing TrajectoryViewer component.

5 — New Database Models

Two new tables added to the existing PostgreSQL schema. No changes to existing tables.

-- New table: session_shares
CREATE TABLE session_shares (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id      UUID NOT NULL REFERENCES chat_sessions(session_id),
    slug            VARCHAR(64) UNIQUE NOT NULL,   -- "bananarepublic"
    share_token     VARCHAR(256) NOT NULL,          -- HMAC-signed token
    created_by      INTEGER NOT NULL REFERENCES users(id),
    permissions     VARCHAR(20) DEFAULT 'view',   -- 'view' | 'interact'
    is_active       BOOLEAN DEFAULT TRUE,
    expires_at      TIMESTAMPTZ,                    -- NULL = no expiry
    created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_shares_slug ON session_shares(slug);
CREATE INDEX idx_shares_session ON session_shares(session_id);

-- New table: session_share_viewers (analytics/audit)
CREATE TABLE session_share_viewers (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    share_id        UUID NOT NULL REFERENCES session_shares(id),
    viewer_ip       VARCHAR(45),                    -- IPv4 or IPv6
    viewer_token    VARCHAR(256) NOT NULL,          -- temp token for this viewer
    connected_at    TIMESTAMPTZ DEFAULT now(),
    disconnected_at TIMESTAMPTZ                     -- set on WS close
);

The corresponding SQLAlchemy models go in backend/models.py following the existing patterns (ChatSession, SessionRuntime, etc.).

Hub Changes (Minimal)

# backend/realtime/ws_hub.py — additions

class SessionWebSocketHub:

    async def register(self, session_id: str, ws: WebSocket,
                       read_only: bool = False) -> None:
        # Existing logic + store read_only flag
        self._connections[session_id].append({
            "ws": ws,
            "read_only": read_only
        })

    def is_read_only(self, session_id: str, ws: WebSocket) -> bool:
        # Used by message handlers to reject commands from viewers
        for conn in self._connections.get(session_id, []):
            if conn["ws"] == ws:
                return conn["read_only"]
        return True

9 — Security Considerations

Token Isolation

Share tokens are completely separate from auth tokens. A viewer token grants zero access to the owner's session, account, or other sessions. It only permits reading from a specific session's WebSocket hub.

Read-Only WebSocket

If a viewer sends EXEC_START, CHAT_MESSAGE, EXEC_STDIN, or any write-type message, the backend silently drops it. The read_only flag on the hub connection enforces this.

Instant Revocation

Owner calls DELETE /sessions/{id}/share. Backend sets is_active = false and disconnects all active viewer WebSockets immediately. The slug URL returns 404 from that point.

Optional Expiry

The expires_at column enables time-limited sharing. Backend checks expiry on both the GET /share/{slug} and WS connection. Expired shares return 410 Gone.

Slug Entropy

Slugs must be unguessable. Use cryptographically random slugs (e.g., secrets.token_urlsafe(16) produces 22-char base64 slugs with 128 bits of entropy). Human-readable slugs like "bananarepublic" are optional aliases set by the user.

Rate Limiting

Rate limit GET /share/{slug} per IP (e.g., 30 req/min) to prevent slug enumeration. Rate limit WS connections per share (e.g., max 50 concurrent viewers per session).
