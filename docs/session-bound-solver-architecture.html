SESSION-BOUND 3-MODE SOLVER

Controller + Sandbox Workflow Architecture

This page explains how to make the solver truly bound to a user session while preserving the strict Architect -> Tester -> Coder sequence and SSE-compatible streaming.

What Exists Today

  • Sequential mode orchestrator already runs Architect -> Tester -> Coder in order.
  • Sandbox command execution already goes through internal WS broker.
  • Session DB already stores mode and completion metadata.
usable now partial needs redesign

Main Gaps

  • SSE depends on controller memory, so state is fragile across restarts.
  • Execution tracking can be duplicated between entry route and orchestrator writes.
  • No single durable event log as the source of truth for replay/resume.

Session-Binding Checklist

session ownership check mode sequencing issue/pr lifecycle marks sandbox command relay

Missing durable run/event records is the key reason the system feels "not fully connected" to the session lifecycle.

Target Architecture (Source of Truth = DB + Event Log)

  • session_pipeline_runs: one active pipeline per session, with idempotency key.
  • agent_executions: one row per mode, linked to pipeline run and predecessor mode.
  • session_stream_events: append-only event log consumed by SSE.
  • Controller SSE endpoint streams from persisted events, not in-memory task maps.

Hard Rules

  • Tester requires architect issue metadata.
  • Coder requires architect issue + tester branch metadata.
  • Any mode failure transitions pipeline to failed and stops next mode.

SSE Compatibility Strategy

  • Keep existing public SSE path stable during migration.
  • Back it with persisted event rows and optional controller relay from sandbox events.
  • Add reconnect cursor (`after_seq`) to recover events after network drop.
  • Cut over frontend to tunnel/session-targeted stream once stable.

Execution Plan (Safe Incremental Cutover)

Step 1

Add durable pipeline + stream event tables and enforce single active run per session.

Step 2

Refactor execution start endpoint to create pipeline run once and hand off to orchestrator idempotently.

Step 3

Persist every mode and progress event as append-only event log records.

Step 4

Serve SSE from event log, keep existing endpoint contract, add reconnect cursor support.

Step 5

Switch frontend stream target from controller-only path to runtime/tunnel-aware path.

Step 6

Retire controller in-memory trajectory stream path after verification and soak period.

Operational Readiness

  • Idempotency key required on execution start API.
  • Session-level DB lock on pipeline start/resume.
  • Audit events for mode start/complete/fail and lifecycle side effects.
  • Failure injection tests: controller restart, sandbox restart, mid-mode disconnect.
