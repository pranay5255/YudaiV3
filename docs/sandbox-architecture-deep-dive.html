⬡ YudaiV3 · Phase 3 Internals — v2 (Revised Architecture)

Sandbox Architecture Deep Dive

Revised architecture: single unified sandbox per session, solver endpoints on the controller host, scoped DB access, WebSocket trajectory streaming, async git clone, and sandbox reuse across solves.

1

Single Unified Sandbox (Collapsed Architecture)

Key change from v1: There is now only one Modal sandbox per session. The separate "solver sandbox" has been collapsed into the realtime sandbox. The realtime sandbox runs both the FastAPI server and the mini-swe-agent as a subprocess. Solver endpoints (start/status/cancel) have been moved to the controller host.

🟦 Single Sandbox — Realtime + Solver (Unified)

Modal app: yudai-realtime Long-lived 1 per session Reused across solves

What runs inside:

uvicorn run_sandbox_server:app --port 8100 ├── sandbox_router (healthz, internal WS exec) ├── heartbeat loop (→ controller every 10s) │ └── [on solve start — triggered by controller] python /home/user/run_agent.py (subprocess) 1. git clone {repo} → /home/user/testbed/ (async, non-blocking) 2. mini-swe-agent.DefaultAgent.run(task) → trajectory saved to /home/user/last_mini_run.traj.json 3. on completion: push trajectory events via WS to controller 4. controller creates PR via lifecycle service

Entrypoint: modal.Sandbox.create("python", "-m", "uvicorn", ...)

Image: debian-slim + git + gh + fastapi + uvicorn + mini-swe-agent + backend source

Tunnel: Modal exposes port 8100 over HTTPS → sandbox.tunnels()[8100].url

Lifetime: Stays alive across multiple solves. Terminates on session end or TTL expiry.

DB Access: Scoped Postgres role — only solves, solve_runs tables (INSERT/UPDATE). No access to auth_tokens, users, or other sensitive tables.

🟧 Controller Host — Solver Endpoints (Moved from Sandbox)

Hosted on: controller API No separate sandbox

What changed:

Solve endpoints (start, status, cancel, stream) previously ran inside the realtime sandbox server. They are now mounted on the controller host (run_controller.py) as part of the controller API.

Controller API (run_controller.py): ├── auth_router (/auth/...) ├── github_router (/github/...) ├── session_router (/daifu/...) ├── controller_router (/controller/...) └── solve_router (/sessions/{id}/solve/...) ← MOVED HERE POST /start → resolves context, sends exec cmd to sandbox GET /status → reads Solve/SolveRun from DB POST /cancel → sends cancel signal to sandbox via WS

Why: Moving solve endpoints to the controller eliminates the need for the frontend to resolve tunnel URLs for solve operations. All solve requests go through the controller, which proxies execution commands to the sandbox via the internal WebSocket (Surface 4).

Context resolution: The controller resolves github_token, issue_text, repo_url from the DB before sending the solve command to the sandbox. The sandbox never reads AuthToken or sensitive tables.

Identity System (Unchanged)

Each sandbox is identified by a canonical key derived from org, repo owner, repo name, and environment. This prevents two sessions from spawning duplicate sandboxes for the same repo.

identity_key = "{org}/{repo_owner}/{repo_name}/{environment}" e.g. "yudai/acme/api-service/main" Rules: - lowercase - replace [^a-z0-9._-] → "-" - collapse "--" → "-" - strip leading/trailing separators

Git Clone — Now Async Inside Sandbox

v1: Git clone ran synchronously on the controller host via subprocess.run(), blocking the async event loop.

v2: Git clone runs inside the sandbox as part of the agent script, using asyncio subprocess. The controller thread is never blocked.

v1 (removed): Controller → subprocess.run("git clone ...") ← BLOCKS event loop v2 (current): Sandbox subprocess → asyncio git clone → /home/user/testbed/ Controller thread is free during clone. Repo persists across solves (git fetch + reset for next run).
Bonus: Since the sandbox is reused across solves, the repo is already cloned for subsequent runs. Only a git fetch --all && git reset --hard origin/{branch} is needed.

Controller ↔ Sandbox Connection Surfaces (v2)

v2 changes: Surface 5 (SSE trajectory) is replaced by WebSocket trajectory streaming through the controller's unified WS hub. Surface 3 (shared DB) is scoped down to only solve/solve_run tables. Surface 4 (WS exec) is now the primary channel for triggering solves.

Surface 1 — Sandbox → Controller Heartbeat sandbox-pushevery 10s

The sandbox server runs a background asyncio loop that sends POST {CONTROLLER_BASE_URL}/controller/sandboxes/{SANDBOX_ID}/heartbeat every 10 seconds. The controller updates sandbox.last_heartbeat_at. Stale detection threshold: 600s (10 min).

Unchanged from v1.

Surface 2 — Controller → Sandbox Liveness Probe controller-pullevery 10s

The SandboxManager probes {tunnel_url}/healthz every 10 seconds. On each result it updates sandbox.last_heartbeat_at and transitions status.

Unchanged from v1.

Surface 3 — Scoped PostgreSQL scoped accesssecurity improved

The sandbox receives CONTROLLER_DATABASE_URL with a scoped Postgres role (sandbox_worker) that only has access to solves and solve_runs tables.

Sandbox can write solves (status updates), solve_runs (trajectory_data, status, pr_url)
Sandbox CANNOT read auth_tokens, users, sandboxes, session_runtime, chat_sessions
How sensitive data reaches sandbox Controller resolves github_token, issue_text, repo_url at solve start time and passes them via WS exec command (Surface 4)
Security fix: Even if the sandbox process is compromised, it cannot exfiltrate OAuth tokens or user data. The blast radius is limited to solve/solve_run metadata.
Postgres Role Setup: CREATE ROLE sandbox_worker WITH LOGIN PASSWORD '...' GRANT SELECT, INSERT, UPDATE ON solves, solve_runs TO sandbox_worker REVOKE ALL ON auth_tokens, users, sandboxes, session_runtime FROM sandbox_worker

Surface 4 — Controller → Sandbox WS Exec controller-pushprimary channel

The controller opens an internal WebSocket to wss://{tunnel_url}/internal/sessions/{session_id}/ws/exec to trigger solve execution. This is now the primary mechanism for starting solves (previously, solve/start was called directly on the sandbox server).

DirectionController → SandboxAuth?secret= query param
Used forStarting agent runs, cancelling runs, streaming trajectory backFilesandbox_routes.py:websocket_internal_exec()
BidirectionalController sends exec commands; sandbox sends trajectory events back over the same WS
Note: In v1 this surface was "implemented but not yet called in production flow." In v2, it is the primary solve execution channel.

Surface 5 — SSE Trajectory Stream → WebSocket via Controller Hub REPLACEDWS via controller

v1 (removed): Frontend connected directly to sandbox tunnel URL via SSE (GET {tunnel_url}/sessions/{id}/solve/stream/{solve_id}/{run_id}). Frontend needed to resolve the tunnel URL. SSE polled every 3s.

v2 (current): Trajectory updates flow through the controller's unified WebSocket hub. The sandbox pushes trajectory events to the controller (via Surface 4 WS or internal HTTP POST), and the controller's ws_hub fans out to all connected frontend sockets for the session.

v1: Frontend → (resolve tunnel URL) → SSE on sandbox → 3s poll lag v2: Sandbox → (WS/HTTP) → Controller WS Hub → Frontend WS - No tunnel URL needed by frontend - Push-based: <100ms latency (no polling) - Uses existing useSessionWebSocket hook - trajectory_update events sent via ws_hub.send_to_session()

Latency Map — v2 Improvements

v2 eliminates two major latency sources: the nested solver sandbox cold start (40–120s) and the 3s SSE polling lag. The critical cold path drops from 3–7 minutes to 1–3 minutes. Warm path (sandbox reuse) drops to 5–20 seconds.

Latency Wins (v2 vs v1)

  • Nested sandbox eliminated: No second Modal sandbox creation. Saves 40–120s per solve. Agent runs as subprocess in the unified sandbox.
  • Sandbox reuse: Subsequent solves in the same session skip sandbox provisioning entirely. Repo already cloned — just git fetch + reset.
  • Push-based trajectory: WebSocket replaces 3s SSE polling. Trajectory updates arrive in <100ms.
  • Async git clone: Controller thread is never blocked by git operations. Clone runs inside the sandbox subprocess.
  • No tunnel URL resolution for frontend: Frontend uses the unified WS to the controller for everything — no extra HTTP hop to resolve tunnel URLs.

Remaining Latency Sources

  • Modal cold start: Still 30–90s for the initial sandbox creation. Can be mitigated with pre-warming or keep-alive strategies.
  • Image pull: First time pulling the unified image (now larger with mini-swe-agent included) can take 60–120s. Mitigated by Modal's layer caching.
  • git clone (first solve): Still 10–90s for large repos on first solve. Subsequent solves reuse the cloned repo.
  • DB writes: Heartbeat writes to Postgres every 10s per sandbox. At scale, consider batching or moving to Redis for heartbeats.

Changes from v1 Architecture

Seven architectural changes applied to the v1 design. Each change is being implemented by the development team.

1 — Solver Endpoints Moved to Controller Host

In Progress

v1: Solve endpoints (POST /solve/start, GET /solve/status, POST /solve/cancel, GET /solve/stream) were mounted on the sandbox server (run_sandbox_server.py). The frontend had to resolve the sandbox tunnel URL to call them.

v2: All solve endpoints are mounted on the controller host (run_controller.py). The controller resolves context (GitHub token, issue text, repo URL) from the DB and sends execution commands to the sandbox via the internal WebSocket. The frontend never needs to know the sandbox tunnel URL for solve operations.

solve_routes.py → controller host frontend simplified context resolved server-side

2 — Async Git Clone Inside Sandbox

In Progress

v1: SandboxManager.ensure_git_bootstrap() ran subprocess.run() synchronously on the controller host during create_runtime_for_session(). This blocked the controller's async event loop during the entire git clone.

v2: Git clone runs inside the sandbox as part of the agent script, using async subprocess execution. The controller thread is never blocked. The cloned repo persists across solve runs within the same sandbox.

controller event loop unblocked clone moves to sandbox process repo persists across solves

3 — Scoped DB Access (Least Privilege)

In Progress

v1: The sandbox had full read/write access to the entire database including AuthToken (all users' GitHub OAuth tokens). Sandbox compromise = all tokens leaked.

v2: The sandbox uses a scoped Postgres role (sandbox_worker) with only SELECT, INSERT, UPDATE on solves and solve_runs. Sensitive data (github_token, issue_text) is resolved by the controller and passed to the sandbox via the internal WS exec command at solve start time.

auth_tokens no longer accessible from sandbox context passed via WS exec command v1 was highest security risk — now mitigated

4 — SSE Replaced with WebSocket for Trajectory Streaming

In Progress

v1: Frontend connected to GET {tunnel_url}/sessions/{id}/solve/stream/{solve_id}/{run_id} via SSE. Polling every 3s. Required tunnel URL resolution. Up to 3s lag on every update.

v2: Trajectory updates flow through the controller's unified WebSocket hub. The sandbox pushes trajectory events to the controller via internal WS (Surface 4). The controller's ws_hub.send_to_session() broadcasts to all connected frontend sockets. Latency drops from 3s polling to <100ms push-based.

3s lag → <100ms uses existing ws_hub infrastructure useTrajectoryStream hook deprecated

5 — Collapsed into Single Sandbox (No Nested Solver)

In Progress

v1: Two Modal sandboxes per session — a "realtime" sandbox running the FastAPI server and a "solver" sandbox running mini-swe-agent. The realtime sandbox spawned the solver sandbox via HeadlessSandboxExecutor. Two sequential cold starts (40–120s each).

v2: Single unified sandbox runs both the FastAPI server and mini-swe-agent as a subprocess. The sandbox image includes all dependencies (mini-swe-agent, git, gh CLI). Agent runs as subprocess.Popen("python /home/user/run_agent.py") inside the same sandbox. One cold start instead of two.

saves 40–120s per solve halves Modal billing HeadlessSandboxExecutor simplified to SubprocessExecutor

6 — Frontend Uses Controller WS for Trajectory

In Progress

v1: Frontend used useTrajectoryStream hook with EventSource API to connect directly to the sandbox tunnel URL for SSE trajectory streaming. Required resolving the tunnel URL and managing a separate connection.

v2: Frontend uses the existing useSessionWebSocket hook to receive trajectory updates via the controller's unified WebSocket. The trajectory_update message type is already defined in ws_protocol.py. No separate connection needed. useTrajectoryStream hook is deprecated.

single WS connection for everything useTrajectoryStream.ts → deprecated useSessionWebSocket already handles trajectory_update

7 — Sandbox Reuse Across Solve Runs

In Progress

v1: The realtime sandbox terminated when a PR was created. Each new solve required a fresh sandbox provisioning (cold start).

v2: The sandbox stays alive across multiple solve runs within the same session. After a solve completes (PR created or not), the sandbox remains running. For subsequent solves: the repo is already cloned, the sandbox is warm, and only a git fetch + reset is needed. Sandbox terminates on session end or idle TTL expiry.

subsequent solves: 5–20s instead of 1.5–5min idle billing during TTL window lifecycle.mark_pr_created() no longer terminates sandbox
