Sandbox Architecture Deep Dive

How Modal sandboxes are provisioned, what runs inside them, how they stay connected to the controller API, where latency hides, and what you need to decide before optimising for production.

1

Two Completely Different Sandbox Types

Critical mental model: there are two independently provisioned Modal sandboxes. One runs your web server. One runs the coding agent. They never share a process, filesystem, or network namespace. The solver sandbox is spawned from inside the realtime sandbox.

🟦 Sandbox Type 1 — Realtime (Session Server)

Modal app: yudai-realtime Long-lived 1 per session

What runs inside:

uvicorn run_sandbox_server:app --port 8100 ├── sandbox_router (healthz, internal WS exec) └── solve_router (start / status / cancel / SSE stream)

Entrypoint: modal.Sandbox.create("python", "-m", "uvicorn", ...)

Image: debian-slim + git + fastapi + uvicorn + backend source code copied in at /app/backend/

Tunnel: Modal exposes port 8100 over HTTPS → sandbox.tunnels()[8100].url

Lifetime: Stays alive until issue + PR are both created, or manual terminate

🟧 Sandbox Type 2 — Solver (mini-swe-agent)

Modal app: yudai-solver Short-lived 1 per solve run

What runs inside:

python /home/user/run_agent.py 1. git clone {repo} → /home/user/testbed/ 2. fetch issue text from GitHub API 3. mini-swe-agent.DefaultAgent.run(task) → trajectory saved to /home/user/last_mini_run.traj.json 4. exit 0 on "Submitted" bash /home/user/create_pr.sh (Phase 2, only on success) 1. git checkout -b yudai/fix-issue-{N}-{ts} 2. git add -A && git commit && git push 3. gh pr create → prints PR URL

Image: debian-slim + git + gh CLI + mini-swe-agent installed from GitHub

Created by: HeadlessSandboxExecutor.run() inside the realtime sandbox process

Lifetime: One solve run (30–90 min). Terminates in finally block.

Identity System

Each sandbox is identified by a canonical key derived from org, repo owner, repo name, and environment. This prevents two sessions from spawning duplicate sandboxes for the same repo.

identity_key = "{org}/{repo_owner}/{repo_name}/{environment}" e.g. "yudai/acme/api-service/main" Rules: - lowercase - replace [^a-z0-9._-] → "-" - collapse "--" → "-" - strip leading/trailing separators
Single Active Editor: if a non-terminated sandbox with this identity_key already exists and is attached to a different session, provisioning raises SINGLE_ACTIVE_EDITOR_CONFLICT.

Git Bootstrap — Controller Host Cache

When the controller provisions a sandbox it also runs a git clone/fetch on the controller host filesystem at /home/yudai/.cache/repos/{identity_key}/.

This is separate from the git clone inside the solver sandbox. Its purpose is to pre-warm a local copy on the controller host for context generation (yudai-grep, facts_and_memories).

Status outcomes: "cloned" → first time, full git clone "fetched" → elapsed ≥ 300s, ran git fetch --all --prune "reused" → cache fresh, no network call "skipped" → repo_url missing

Controller ↔ Sandbox Connection Surfaces

There are 5 distinct connection surfaces between the controller API host and the realtime sandbox. Understanding each surface's direction, frequency, failure mode, and latency cost is essential for designing a reliable system.
🏓

Surface 1 — Sandbox → Controller Heartbeat sandbox-pushevery 10s

The sandbox server runs a background asyncio loop that sends POST {CONTROLLER_BASE_URL}/controller/sandboxes/{SANDBOX_ID}/heartbeat every 10 seconds. The controller updates sandbox.last_heartbeat_at. Stale detection threshold: 600s (10 min) — after that, cleanup_stale_sandboxes() terminates it.

DirectionSandbox → ControllerAuthX-Controller-Heartbeat-Secret header
Failure modeSandbox marked STOPPED after missing heartbeatsFilerun_sandbox_server.py:_heartbeat_loop()
Latency budget5s HTTP timeoutRiskController restart resets the stale timer — could falsely terminate healthy sandboxes
🔍

Surface 2 — Controller → Sandbox Liveness Probe controller-pullevery 10s

The SandboxManager runs a probe loop against {tunnel_url}/healthz every 10 seconds using httpx with a 3s timeout. On each result it fires a ProbeCallback that writes back to the DB: updates sandbox.last_heartbeat_at, transitions status PROVISIONING→RUNNING or RUNNING→STOPPED.

Note: Both Surface 1 and Surface 2 update last_heartbeat_at independently. This means heartbeat and probe can interfere — a probe failure does not immediately mean the sandbox is dead if heartbeats are still arriving. Currently there is no de-duplication logic between these two signals.
DirectionController → Sandbox (via tunnel)AuthNone (public /healthz)
Failure modesandbox.status → STOPPED; probe continuesFilesandbox_manager.py:start_probe()
🗄️

Surface 3 — Shared PostgreSQL both read/writehighest risk

The sandbox server receives CONTROLLER_DATABASE_URL as an env var at boot, pointing to the same PostgreSQL instance as the controller. solve_manager.py writes Solve, SolveRun rows and reads ChatSession, Issue, AuthToken, AIModel directly — no HTTP intermediary.

Controller writes Sandbox, SessionRuntime, SessionAuditEvent, SessionArtifact
Sandbox writes Solve, SolveRun (via solve_manager), lifecycle events (via lifecycle service calls)
Failure mode If DB is unreachable from sandbox, solve_manager raises; runs stay PENDING forever
Risk Multiple sandboxes writing to the same DB tables simultaneously — no row-level locking on Solve beyond SQLAlchemy session scope
Security note: The sandbox has full read access to all DB tables including AuthToken (GitHub tokens for all users). If the sandbox process is compromised, it can exfiltrate all OAuth tokens. This is the highest-risk surface in the current architecture.
🔌

Surface 4 — Controller WebSocket → Sandbox WS Exec controller-pushon-demand

The controller can open an internal WebSocket to wss://{tunnel_url}/internal/sessions/{session_id}/ws/exec authenticated by CONTROLLER_INTERNAL_WS_SECRET. This allows the controller to run arbitrary shell commands inside the sandbox and stream stdout/stderr back in real time.

DirectionController → Sandbox (internal)Auth?secret= query param
CommandsArbitrary bash (for mode orchestrator MSWEA)Filesandbox_routes.py:websocket_internal_exec()
StatusImplemented but not yet called in production flowRiskArbitrary code execution if secret leaks
📡

Surface 5 — Sandbox SSE → Frontend sandbox-pushstreaming

Once the frontend resolves the tunnel URL from the controller, it connects directly to the sandbox server for trajectory streaming: GET {tunnel_url}/sessions/{id}/solve/stream/{solve_id}/{run_id}?token=.... The sandbox polls HeadlessSandboxExecutor.read_trajectory() every 3 seconds and pushes SSE events. Auth is a JWT passed as a query param (EventSource cannot set headers).

Auth?token= JWT validated against DBFilesolve_routes.py:stream_trajectory()
Events emittedtrajectory_update, heartbeat, status, done, errorHeartbeatEvery 15s (5 polls)

Latency Map — Where Time Goes

The critical path from "user clicks Start" to "agent is editing code" currently has 4 blocking steps that can add 3–7 minutes of dead time before a single line of code is touched.

Total cold path: 3–7 minutes Warm path (cached image): 45–90s Hot path (pre-warmed sandbox): 5–15s

Worst-Case Bottlenecks

  • Modal image pull: mini-swe-agent is installed from GitHub at image build time. If the image is not cached in Modal's layer cache, the pip install -e /opt/mini-swe-agent runs fresh — 60–120s alone.
  • Nested sandbox creation: The realtime sandbox must boot fully before it can spawn the solver sandbox. Two sequential cold starts compound.
  • git clone per run: Every solve run clones the repo fresh into /home/user/testbed/. Large repos (>500MB) can take 60–90s.
  • Trajectory polling: The SSE endpoint polls every 3s. If the agent finishes in 2.9s, the UI sees nothing for 3s. At high step frequency, the UI is always 0–3s behind.

Structural Latency Sources

  • Two-hop architecture: Browser → Controller → (tunnel resolve) → Sandbox. Every user action that goes via the controller adds a round trip before reaching the sandbox.
  • 3s poll on trajectory: The realtime sandbox polls solver_sandbox.read_file() → Modal's sandbox.open() → file read from solver sandbox filesystem. This is a blocking I/O operation in every poll loop iteration.
  • Synchronous git bootstrap: SandboxManager.ensure_git_bootstrap() calls subprocess.run() synchronously on the controller thread during provisioning. A slow clone blocks the controller's async event loop.
  • DB writes on critical path: Every heartbeat writes to Postgres. At 10s intervals with N active sandboxes, this is O(N) DB writes/s from the controller alone.

Improvement Recommendations

1 — Pre-warm Solver Sandbox Image (Warm Pool)

High Impact

The single biggest latency win. Instead of creating the solver Modal sandbox on-demand when solve/start is called, maintain a small warm pool of pre-created solver sandboxes with mini-swe-agent already booted, sitting idle. When a solve starts, claim one from the pool and replenish asynchronously.

Current: solve/start → modal.Sandbox.create() → wait 40–120s → upload script → execute With pool: solve/start → claim from pool (< 1s) → upload script → execute background task replenishes pool to target size (e.g. 2 warm sandboxes)

Implementation: A SolverSandboxWarmPool singleton in the realtime sandbox server. On boot, pre-create N sandboxes. Expose pool size in /healthz.

saves 40–120s per solve Modal cost: idle sandboxes still billed target pool size: 1–3

2 — Pre-clone Repo at Sandbox Boot (Shared Workspace)

High Impact

Currently the solver sandbox clones the repo fresh every run. Instead, the realtime sandbox should clone the repo at boot time into a shared volume or pass a pre-cloned tarball into the solver sandbox at creation time. Modal supports mounting volumes that persist across sandbox invocations.

Option A — Volume mount: realtime sandbox boots → git clone repo → Modal.Volume("/workspace") solver sandbox created with same volume → skip clone, just git reset --hard HEAD Option B — Tarball injection: realtime sandbox boots → git archive HEAD → upload tarball to solver sandbox solver sandbox: tar -xzf repo.tar.gz → /home/user/testbed/ (no git history needed)
saves 10–90s per solve requires Modal volume setup or tarball upload step

3 — Push-based Trajectory Streaming via WS (replace 3s poll)

High Impact

The current SSE endpoint polls executor.read_trajectory() every 3s. This means up to 3s lag on every trajectory update and a constant file-read overhead. Replace with push-based streaming: mini-swe-agent writes each step to the trajectory file and sends a signal (via a Unix socket, named pipe, or small HTTP call to localhost) that tells the sandbox server to immediately read and push the new step.

Option A — Named pipe / Unix socket: run_agent.py: after each step → write to /tmp/traj_events.pipe sandbox server: asyncio.open_unix_connection() → read each step → push to SSE/WS immediately Option B — Patch DefaultAgent (if open source allows): Override agent.save() → call localhost HTTP endpoint after writing trajectory Option C — inotify on the trajectory file: Use watchfiles/aionotify to watch /home/user/last_mini_run.traj.json → push on change
reduces trajectory lag: 3s → <100ms removes constant file-read overhead requires modifying agent script or using inotify

4 — Move git Bootstrap off the Controller Thread

Medium Impact

SandboxManager.ensure_git_bootstrap() calls subprocess.run() synchronously during create_runtime_for_session(), which runs in an async FastAPI handler. A large git clone blocks the controller's event loop for the duration of the clone. Wrap it in asyncio.to_thread() or dispatch to a background task.

Current: await lifecycle.create_runtime_for_session() ← blocks on git clone Fixed: asyncio.to_thread(sandbox_manager.ensure_git_bootstrap, ...) or: return early, clone happens in background
unblocks event loop during provisioning 1-line fix: asyncio.to_thread

5 — Decouple Heartbeat + Probe Signals

Medium Impact

Currently both S1 (sandbox heartbeat) and S2 (controller probe) update last_heartbeat_at. This means a single field tracks two different health signals. If the sandbox is alive (heartbeats arriving) but the tunnel is down (probes failing), the sandbox appears healthy. Separate these into two fields: last_heartbeat_at and last_probe_at. Use compound logic for stale detection: stale if both signals are absent.

more accurate liveness tracking DB migration: add last_probe_at column

6 — Scope DB Access from Sandbox (Least Privilege)

High Impact (Security)

The sandbox currently has full read access to all DB tables including AuthToken (all users' GitHub OAuth tokens). Apply least-privilege: either use a scoped Postgres role that only has access to solves, solve_runs, and the specific chat_sessions / issues rows it needs — or proxy all sensitive reads through a controller API endpoint that validates ownership before returning data.

Option A — DB role scoping: CREATE ROLE sandbox_worker WITH LOGIN PASSWORD '...' GRANT SELECT ON chat_sessions, issues, ai_models TO sandbox_worker GRANT SELECT, INSERT, UPDATE ON solves, solve_runs TO sandbox_worker (no access to auth_tokens, users, other users' data) Option B — Data passed at provisioning time: Controller resolves github_token, repo_url, issue text at solve/start time Passes them in the solve request body — sandbox never touches auth_tokens table
current: sandbox can read all auth_tokens Option B is simpler and eliminates the risk entirely

7 — Replace SSE with WebSocket for Trajectory (Unified Stream)

Medium Impact

The unified WS endpoint (ws_hub.py) already exists on the controller. The frontend's useSessionWebSocket hook already has reconnect logic. Consolidate trajectory streaming into this channel: the sandbox server pushes trajectory events to the controller's WS hub via an internal HTTP call or by writing to a shared Redis pub/sub channel, and the controller fans out to connected frontend sockets. This removes the need for the frontend to directly connect to the sandbox tunnel URL.

Current: Frontend → (tunnel URL) → SSE on sandbox server Target: Frontend → (WS) → Controller WS hub Sandbox → (internal HTTP) → Controller "push trajectory event" endpoint Controller WS hub broadcasts to all connected frontend sockets for this session
frontend no longer needs the tunnel URL requires a controller-side trajectory event endpoint enables WS reconnect without losing session context

Critical Design Questions

These are the architectural decisions that must be made before implementing the low-latency optimisations. Each one has a significant downstream impact on the shape of the system. Answer these to lock down the design.

1Where does the coding agent actually execute — inside the realtime sandbox or in a separate nested solver sandbox?

Right now: the realtime sandbox (FastAPI server) creates a second Modal sandbox to run mini-swe-agent. This means two Modal sandboxes are alive simultaneously per session.

  • Keep nested: Clean separation — the server and the agent can't interfere. But adds 40–120s cold start and doubles Modal cost.
  • Collapse into one: The realtime sandbox runs mini-swe-agent directly as a subprocess. Faster, cheaper, but the agent process can kill the server.
  • Hybrid: Pre-warm nested solver sandboxes so the creation cost is paid upfront.
Keep nested (current)
Collapse into one sandbox
Keep nested + warm pool
2How should the frontend get trajectory updates — SSE from the sandbox tunnel, or WS from the controller?

This determines the frontend's connection model and whether it ever needs to know the sandbox tunnel URL.

  • SSE from tunnel (current): Frontend must resolve tunnel URL, connect directly to sandbox. Simple. But frontend needs tunnel access, and SSE reconnects don't recover session state.
  • WS from controller: Frontend has one stable WS connection to the controller. Sandbox pushes events to controller via internal HTTP. Controller fans out. More resilient to sandbox restarts.
  • Hybrid: WS for chat/status events (controller), SSE for raw trajectory bytes (sandbox). Separates concerns but adds complexity.
SSE direct from sandbox
WS via controller hub
Hybrid WS+SSE
3Should the sandbox have direct database access, or should all sensitive reads go through the controller API?

This is a security vs. simplicity trade-off. Currently the sandbox reads AuthToken, User, and other sensitive tables directly.

  • Direct DB (current): Fastest. Simple code. But sandbox can read all auth tokens. Risk: sandbox compromise = all GitHub tokens leaked.
  • Proxy through controller: Sandbox calls POST /controller/sessions/{id}/solve/resolve-context to get pre-resolved {github_token, issue_text, repo_url}. No direct DB access. Requires one more HTTP round-trip at solve start.
  • Scoped DB role: Middle ground — sandbox gets a Postgres role with only the tables/rows it needs. Requires DB migration and role management.
Direct DB (current)
Proxy via controller API
Scoped Postgres role
4What is the repo delivery strategy for the solver sandbox — clone at runtime, pre-clone at provisioning, or volume mount?

The git clone is currently the largest variable-latency operation (10–90s depending on repo size).

  • Clone at runtime (current): Simple. Always fresh. But adds 10–90s to critical path every solve.
  • Clone at realtime sandbox boot: Realtime sandbox clones on boot, passes as a tarball or mounts via Modal volume. Saves time on solve start but requires workspace management.
  • Shallow clone + filter: Use --depth 1 --filter=blob:none to minimise clone size. Fast even for large repos.
  • Modal Volume: Persistent volume shared between solver sandbox runs for the same identity key. First clone is slow, subsequent runs get git reset --hard HEAD.
Clone at runtime (current)
Shallow clone (--depth 1)
Pre-clone at boot + tarball
Modal Volume
5How should sandbox-to-controller communication work when the controller is unavailable — fail hard or buffer?

If the controller goes down during a solve, the sandbox continues running but heartbeats fail. The controller's stale detection would terminate the sandbox row after 600s even if the agent is mid-solve.

  • Fail hard (current implicit): No retry logic on heartbeats beyond the interval. Controller restart marks sandbox stale.
  • Buffer heartbeats: Sandbox retries heartbeats with exponential backoff. Controller distinguishes "missed heartbeats" from "terminated".
  • Self-contained solve: Sandbox completes the solve regardless of controller availability, writes results to DB, controller reconciles on recovery.
Fail hard + user retries
Buffer + retry heartbeats
Self-contained solve
6What is the target latency from "user clicks Solve" to "first agent step visible in UI"?

This shapes everything from warm pool sizing to whether you need pre-provisioning at session creation time. Different targets require fundamentally different architectures.

  • <5 seconds: Requires warm pools, pre-cloned repos, and push-based streaming. Significant infrastructure investment.
  • 5–30 seconds: Achievable with warm pool + shallow clone + push streaming. Moderate investment.
  • 30–120 seconds: Achievable with just shallow clone + better progress UX. Low investment.
  • >2 minutes acceptable: Current architecture with minor improvements is fine.
<5 seconds (enterprise)
5–30 seconds (fast)
30–120 seconds (standard)
>2 min OK for now
7Should one realtime sandbox be reused across multiple solve runs for the same session, or create a fresh one per run?

Currently a realtime sandbox terminates when a PR is created. If the user wants to run another solve, a new sandbox must be provisioned (cold start again). Reuse would amortise the cold-start cost.

  • Terminate on completion (current): Clean slate per solve. But cold start on every new solve within the same session.
  • Keep alive between solves: Faster for multi-solve sessions. But idle billing and state management complexity (stale workspace).
  • Keep alive with TTL: Reuse for N minutes after last solve. If user comes back within TTL, instant start. After TTL, terminate.
Terminate on completion (current)
Keep alive between solves
Keep alive with TTL
Recommended decision order: Answer Q6 first (target latency), then Q1 (one vs. two sandboxes), then Q4 (repo delivery), then Q3 (DB access security), then Q2 (streaming architecture), then Q5 + Q7 (resilience). The answer to Q6 collapses most of the others.
