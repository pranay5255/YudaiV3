Section 01

Core Networking Concepts

Before looking at the project specifically, here are the building blocks you need to understand. Everything else in this document is built on top of these six ideas.

🌐

IP Address

A unique number that identifies a machine on the internet. Like a postal address. 139.84.154.9 is the public IP of your backend server. Every packet of data on the internet is routed to an IP address.

📛

Domain Name

A human-readable label like api.yudai.app that maps to an IP address. Browsers use the DNS system to translate names → numbers before connecting.

🚪

Port

A numbered "door" on a server. One IP can have 65 535 ports. Port 443 = HTTPS, port 80 = HTTP, port 5432 = PostgreSQL. You always connect to IP:port.

🔒

TLS / HTTPS

An encryption layer wrapped around HTTP. Data is scrambled in transit so that no one between you and the server can read it. Required for login, cookies, and APIs.

↔️

Reverse Proxy

A program that sits in front of your app and forwards requests to it. Clients talk to the proxy; the proxy talks to your app. Caddy is your reverse proxy.

📦

Container

A lightweight, isolated process that bundles your app and all its dependencies. Like a virtual machine but much lighter. Docker manages your containers.

Section 02

How DNS Works

DNS (Domain Name System) is the internet's phone book. When your browser wants to connect to api.yudai.app, it first has to find out the IP address behind that name.

DNS lookup flow
  Browser types:  https://api.yudai.app/auth/api/user

  1. Browser asks local DNS resolver (your ISP or 8.8.8.8):
     "What IP is api.yudai.app?"

  2. Resolver walks the DNS tree:
     Root nameserver  →  .app nameserver  →  yudai.app nameserver
                                                      (your domain registrar)

  3. Your registrar returns the A record:
     api.yudai.app  A  139.84.154.9

  4. Browser now connects to:
     139.84.154.9:443

DNS Record Types

Records are key-value pairs stored at your domain registrar (Cloudflare, Namecheap, etc.).

Record Type What it does Example in this project
A Maps a hostname to an IPv4 address api.yudai.app → 139.84.154.9
A Maps root domain to Vercel's IP yudai.app → 76.76.21.21
CNAME Alias that points to another hostname www.yudai.app → cname.vercel-dns.com
AAAA Like A but for IPv6 addresses Not used here
MX Mail exchange — where email goes Not used here
TXT Arbitrary text, used for domain verification Vercel ownership verification
Why "api" as a subdomain? The root domain yudai.app points to Vercel (the frontend). The api subdomain points to your own server at 139.84.154.9. This way one domain name serves two completely different machines.
DNS propagation takes time After you add or change a DNS record, it can take anywhere from 1 minute to 48 hours for all DNS resolvers around the world to see the change. During this window some users may get the old answer and some the new one.

Every network connection goes to a specific port number on a server. Think of an IP address as a building and ports as the individual unit numbers inside.

Section 04

HTTPS & TLS (SSL)

Browsers and APIs communicate over HTTP (HyperText Transfer Protocol). HTTP is plain text — anyone on the network between you and the server can read it. HTTPS wraps HTTP inside TLS (Transport Layer Security), which encrypts the data.

HTTP vs HTTPS
  Without HTTPS (plain HTTP):
  ─────────────────────────────────────────────────────────────────
  Browser  ──── GET /auth/api/user ──────────────────────►  Server
           ◄─── {"token": "abc123", "user": "you@..."} ──
                ↑
                Anyone between you and the server can read this

  With HTTPS (TLS encrypted):
  ─────────────────────────────────────────────────────────────────
  Browser  ──── 🔒 encrypted blob ──────────────────────►  Server
           ◄─── 🔒 encrypted blob ──────────────────────
                ↑
                Looks like random noise to anyone intercepting it

How TLS Certificates Work

To prove a server is who it says it is, it presents a certificate — a digitally signed document that says "I am really api.yudai.app". The signature comes from a trusted Certificate Authority (CA) like Let's Encrypt.

1

Caddy asks Let's Encrypt for a certificate

This happens automatically when Caddy first starts with a domain name configured. It contacts acme-v02.api.letsencrypt.org over HTTPS.

2

Let's Encrypt issues a challenge

Let's Encrypt needs to verify you actually control api.yudai.app. It performs a TLS-ALPN-01 challenge: it connects to port 443 of your server and expects a specific response. Caddy handles this automatically.

3

Certificate is issued and stored

Once verified, Let's Encrypt signs and issues a certificate stored in /var/lib/caddy/.local/share/caddy/. In the logs you saw: "certificate obtained successfully, issuer: Let's Encrypt E7".

4

Caddy auto-renews before expiry

Let's Encrypt certificates last 90 days. Caddy silently renews them in the background ~30 days before expiry. You never have to think about it again.

Your certificate details Subject: CN=api.yudai.app · Issuer: Let's Encrypt E7 · Protocol: TLS 1.3 · Cipher: AES_128_GCM_SHA256
Section 05

What a Reverse Proxy Does

Your FastAPI app runs inside Docker and only knows how to speak plain HTTP on port 8000. It has no concept of domain names, SSL certificates, or routing multiple domains. A reverse proxy (Caddy) sits in front of it and handles all of that.

Caddy as reverse proxy
  ┌─────────────────────────────────────────────────────────────────────┐
  │  SERVER  139.84.154.9                                               │
  │                                                                     │
  │  ┌───────────────────────────────────────────────────────────────┐  │
  │  │  CADDY                                                        │  │
  │  │   • Listens on :80 and :443                                   │  │
  │  │   • Holds the TLS certificate for api.yudai.app               │  │
  │  │   • Decrypts incoming HTTPS traffic                           │  │
  │  │   • Forwards plain HTTP to localhost:8000                     │  │
  │  │   • Re-encrypts response back to client                       │  │
  │  └──────────────────────────┬────────────────────────────────────┘  │
  │                             │ http://localhost:8000                  │
  │  ┌──────────────────────────▼────────────────────────────────────┐  │
  │  │  DOCKER: yudai-be                                             │  │
  │  │   • FastAPI / Uvicorn                                         │  │
  │  │   • Receives plain HTTP, knows nothing about TLS              │  │
  │  │   • Handles /auth, /daifu/sessions, /health, etc.             │  │
  │  └───────────────────────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────────────────────┘

  Internet sees:   HTTPS api.yudai.app:443  (encrypted, trusted cert)
  Caddy passes:    HTTP  localhost:8000      (plain, internal only)

Why not just expose port 8000 directly?

Technically possible, but:

  • You'd serve unencrypted HTTP — browsers block mixed content (HTTPS page → HTTP API)
  • You'd need your app itself to handle SSL certificates — complex, non-standard
  • Port 8000 in URLs looks unprofessional (api.yudai.app:8000)
  • Security headers, rate limiting, logging are easier in a dedicated proxy layer

Your Caddyfile — explained line by line

api.yudai.app {
    reverse_proxy localhost:8000
}
This 3-line file is everything Caddy needs. It implicitly:
• Listens on both port 80 and 443
• Redirects all HTTP → HTTPS automatically
• Obtains and renews a Let's Encrypt cert for api.yudai.app
• Forwards all HTTPS traffic to localhost:8000
Section 06

Docker Networking

Docker creates isolated environments called containers. Each container has its own filesystem, process space, and network interface. Containers on the same Docker network can reach each other by name, but are isolated from the host and internet by default.

Docker network: yudai-network (bridge)
  ┌────────────────────────────────────────────────────────────────────────┐
  │  HOST MACHINE (139.84.154.9)                                           │
  │                                                                        │
  │   PORT MAP (docker-compose "ports"):                                   │
  │   host:8000  ◄──────────────────────────────  container:8000          │
  │   (makes yudai-be reachable from host as localhost:8000)               │
  │                                                                        │
  │  ┌───────────────── yudai-network (172.19.0.0/16) ─────────────────┐  │
  │  │                                                                  │  │
  │  │  ┌──────────────────────────┐   ┌──────────────────────────┐   │  │
  │  │  │  yudai-be                │   │  yudai-db                 │   │  │
  │  │  │  172.19.0.3              │   │  172.19.0.2               │   │  │
  │  │  │                          │   │                            │   │  │
  │  │  │  FastAPI on :8000        │   │  PostgreSQL on :5432       │   │  │
  │  │  │                          │   │                            │   │  │
  │  │  │  Talks to DB via:        │   │  NOT exposed to host       │   │  │
  │  │  │  yudai-db:5432           │◄──┤  or internet               │   │  │
  │  │  └──────────────────────────┘   └──────────────────────────┘   │  │

  │  └──────────────────────────────────────────────────────────────────┘  │
  └────────────────────────────────────────────────────────────────────────┘

Key Docker networking concepts

ConceptWhat it meansIn this project
Bridge network A private virtual network on the host. Containers on it can talk to each other. yudai-network — both containers join it
Container DNS Containers resolve each other by service name, not IP. yudai-be connects to yudai-db:5432
Port mapping host_port:container_port — makes a container port reachable from the host. "8000:8000" → localhost:8000 reaches the container
No port mapping = invisible If no port mapping is defined, the port is only reachable inside the Docker network. yudai-db:5432 — no host mapping, DB stays private
Volumes Persistent storage that survives container restarts. postgres_data stores DB files; app_logs stores logs
healthcheck Docker periodically runs a command to verify the container is working. curl http://localhost:8000/health every 15 seconds
depends_on One container waits for another to be healthy before starting. yudai-be waits for yudai-db to be healthy

Why the DB port is never exposed to the internet

The yudai-db container has no ports: mapping in docker-compose. This means port 5432 only exists inside yudai-network. yudai-be can reach it (same network), but nothing outside the server can. This is intentional — databases should never be directly reachable from the internet.

Request Trace — What Happens When You Log In

Let's trace exactly what happens when a user opens yudai.app and clicks the GitHub login button. Every hop is shown.

Full login request flow
  USER OPENS  https://yudai.app
  ──────────────────────────────────────────────────────────────────────

  1. DNS  browser asks: "what IP is yudai.app?"
          answer: 76.76.21.21  (Vercel)

  2. HTTP  browser connects to 76.76.21.21:443 (Vercel CDN)
          Vercel serves: index.html + index.js (React bundle)

  3. JS    React app boots in browser.
          App calls GET https://api.yudai.app/auth/api/user
          to check if user is already logged in.

  4. DNS  browser asks: "what IP is api.yudai.app?"
          answer: 139.84.154.9  (your server)

  5. TLS   browser connects to 139.84.154.9:443
          Caddy presents cert: CN=api.yudai.app (Let's Encrypt)
          Browser verifies cert is valid and trusted.
          TLS 1.3 handshake completes. Channel is encrypted.

  6. HTTP  Caddy receives: GET /auth/api/user
          Caddy forwards: GET http://localhost:8000/auth/api/user

  7. API   FastAPI receives request.
          No auth token → returns 401 Unauthorized.

  8. UI    React sees 401 → shows Login page.

  ──────────────────────────────────────────────────────────────────────
  USER CLICKS "Login with GitHub"
  ──────────────────────────────────────────────────────────────────────

  9. JS    React calls GET https://api.yudai.app/auth/api/login
          FastAPI returns: { "login_url": "https://github.com/login/oauth/..." }

  10. JS   Browser redirects to GitHub OAuth page.
          User approves.

  11. GH   GitHub redirects to:
          https://api.yudai.app/auth/callback?code=abc123

  12. API  FastAPI receives callback with code
          Calls GitHub API to exchange code → access token
          Creates/updates user in PostgreSQL (yudai-db)
          Creates session token (stored in session_tokens table)

  13. API  FastAPI redirects to frontend:
          https://yudai.app/auth/success?session_token=xyz...

  14. JS   React /auth/success page extracts session_token
          Stores session_token in memory/localStorage
          All future requests include: Authorization: Bearer xyz...

  15. ✅   User is logged in.
Section 09

CORS — Why It Exists

CORS (Cross-Origin Resource Sharing) is a browser security policy that blocks web pages from making API calls to a different domain than the one they were loaded from — unless the server explicitly allows it.

Why CORS is needed
  WITHOUT CORS protection:
  ─────────────────────────────────────────────────────────────────────
  User visits evil.com  →  evil.com's JS silently calls api.yudai.app
  with the user's cookies/tokens  →  steals data.  ❌

  WITH CORS (browser enforces):
  ─────────────────────────────────────────────────────────────────────
  Browser checks: is "https://yudai.app" in the server's allowed origins?

  FastAPI returns header:
  Access-Control-Allow-Origin: https://yudai.app

  Browser sees yudai.app is allowed → permits the request.  ✅
  Browser sees evil.com is NOT listed → blocks the request.  🛡️

Your CORS configuration

Defined in backend/.env.prod and read by FastAPI's middleware:

# backend/.env.prod
ALLOW_ORIGINS=https://yudai.app
ALLOW_ORIGIN_REGEX=^https://.*\.vercel\.app$   # allows preview deployments
FRONTEND_URL=https://yudai.app
GITHUB_REDIRECT_URI=https://api.yudai.app/auth/callback
Common CORS mistake If ALLOW_ORIGINS doesn't exactly match the frontend origin (including https:// and no trailing slash), the browser will block all API calls with a CORS error. The regex pattern also allows Vercel preview URLs like https://yudaiv4-abc123.vercel.app.

Key commands

# Check all running containers and their health
docker ps

# Live backend logs
docker logs yudai-be -f

# Caddy status and cert acquisition logs

# Verify DNS resolution

# Test HTTPS end-to-end (checks cert + connectivity)
curl -sv https://api.yudai.app/health

# Check what's listening on ports
ss -tlnp | grep -E '80|443|8000'

ufw status

# Restart backend without downtime

# Rebuild and redeploy backend
docker compose --env-file backend/.env.prod -f docker-compose.backend-only.yml up -d --build
