01 The Memory Problem

The Blank Slate Problem

🗒️

The Session — The Desk

While a conversation is active, all messages live in the LLM’s context window. But the context window is finite. When the session ends, everything on the desk is gone.

Out of the box, AI agents are stateless. Every turn is just an increasingly long context window. There is no memory between calls.

🗄️

Long-Term Memory — The Filing Cabinet

The filing cabinet persists across sessions. Anything filed stays available in future conversations. Anything not filed is gone when the desk is cleared.

The hard part isn’t the storage. It’s knowing when to write and when to load. Get that right, and the rest is just files.

04 Implementation

Four Mechanisms

1

Bootstrap Loading implemented

At the start of every chat turn, stored memories are injected into the LLM context — even before new F&M are generated. The agent always starts with prior knowledge, not a blank slate.

ChatOps._bootstrap_memory_context()
→ reads session.repo_context["facts_and_memories"]
→ formats as [INTERNAL_FACTS_MEMORIES] block
→ injected into every chat turn unconditionally
2

Accumulation / Consolidation implemented

Two-layer consolidation: first, existing memories are fed back into the LLM prompt so it can deduplicate, update contradictions, and merge intelligently (CONSOLIDATE/DEDUPLICATE/UPDATE/RETAIN/DROP instructions). Then a deterministic MemoryService.accumulate() pass enforces hard caps and catches remaining near-duplicates via 60-char prefix fingerprint. New entries win on collision.

Layer 1: LLM consolidation
→ existing_memories passed to FactsAndMemoriesService.generate()
→ LLM deduplicates, updates contradictions, merges

Layer 2: Deterministic safety net
→ MemoryService.accumulate(existing, new_*)
→ dedup via prefix fingerprint (60 chars)
→ new entry wins on collision
→ capped: facts 30, memories 20, highlights 15
3

Session Snapshot implemented

When a session is deactivated (is_active True→False), the last N meaningful messages are captured as a snapshot — filtering out short messages (<10 chars) and system/tool messages. Like OpenClaw’s /new hook.

MemoryService.save_session_snapshot(db, session)
→ triggered on PUT /sessions/{id} when is_active → False
→ filters: user/assistant only, len > 10 chars
→ saves last 15 meaningful turns to repo_context["session_snapshot"]
→ MemoryService.get_session_snapshot() retrieves it
4

"Remember This" implemented

Explicit memory generation is controlled per-session by a boolean flag set at creation time. When enabled, every chat turn triggers F&M generation and accumulation automatically.

ChatSession.generate_facts_memories (bool)
→ True  → generate + accumulate each turn
→ False → bootstrap from stored memories only
05 What Changed

Before vs After

Before

overwrite bug
  • Every chat turn called stored.update({...}) — completely overwriting prior facts with whatever was extracted from just the latest messages
  • No bootstrap: LLM started each turn with no prior memory, even if rich memories were stored in repo_context
  • Turn 10 → newly extracted facts silently erased everything learned in turns 1–9
  • Agent felt amnesiac: had to re-discover repo structure on every turn
  • No API to inspect what the agent "knows" — black-box debugging

After

accumulation
  • Two-layer consolidation: LLM-based merge (CONSOLIDATE/DEDUP/UPDATE/RETAIN/DROP) + deterministic MemoryService.accumulate() safety net
  • Bootstrap loading via _bootstrap_memory_context() injects prior memories into every turn's LLM context
  • Prefix fingerprint deduplication prevents "User prefers dark mode" × 20 cluttering the memory
  • Collision resolution: new entry wins, so user preferences can be cleanly updated over time
  • Session snapshots (M3): last 15 meaningful messages saved on deactivation via MemoryService.save_session_snapshot()
  • GET /sessions/{id}/memories exposes structured memory for frontend display and debugging
